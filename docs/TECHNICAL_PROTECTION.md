# Техническая защита данных

Краткое описание мер технической защиты, реализованных в боте.

## Шифрование базы данных

- **Модуль:** `database.py`
- **Варианты:**
  1. **SQLCipher** (`pysqlcipher3`) — шифрование файла БД целиком. Требует C++ компилятор при установке (на Windows может быть неудобно).
  2. **Шифрование данных (cryptography)** — шифрование чувствительных полей при записи/чтении. Рекомендуется, работает на Windows без дополнительных сборок.

- **Включение:** в `.env` задать переменную `DB_ENCRYPTION_KEY` (рекомендуется не менее 32 символов).
- **Зависимости:**
  - для шифрования полей: `pip install cryptography`
  - для шифрования файла БД: `pip install pysqlcipher3`
- Если установлен `cryptography` и ключ не задан, приложение при старте выдаст ошибку и не запустится (данные не должны храниться незашифрованными при наличии возможности шифрования).

## Хеширование паролей

- **Модуль:** `security.py`, `database.py`
- Пароли **не хранятся** в открытом виде. Используется **PBKDF2-HMAC-SHA256** (100 000 итераций, соль — `user_id`).
- В БД сохраняется только хеш (`user_passwords.password_hash`). При проверке введённый пароль хешируется и сравнивается с сохранённым хешем.
- Функции: `hash_password`, `save_user_password`, `check_user_password` в `security.py`; в БД — `save_user_password_hash`, `get_user_password_hash`, `check_user_password`.

## Регулярные бэкапы

- **Модуль:** `backup.py`, планирование в `main.py`
- **Расписание:** бэкап выполняется **каждые 24 часа** (86400 секунд) через `job_queue.run_repeating(periodic_backup, ...)`.
- **Директория:** по умолчанию `backups/` в корне проекта. Имя файла: `cognitive_diary_backup_YYYYMMDD_HHMMSS.db`.
- **Шифрование бэкапа:** если задан `DB_ENCRYPTION_KEY`, созданный файл бэкапа шифруется тем же алгоритмом (Fernet на базе PBKDF2), незашифрованный файл удаляется.
- **Хранение:** хранятся последние **10** бэкапов; более старые автоматически удаляются (`cleanup_old_backups`).
- При удалении данных пользователя (`delete_user_data`) или полной очистке (`delete_all_users`) бэкапы в `backups/` **не удаляются** — их можно использовать для восстановления.

## Логи подтверждений (возраст, чувствительная запись)

- Подтверждение возраста при `/start` и подтверждение перед первой чувствительной записью логируются в таблицу **`confirmation_logs`** (`database.py`): `user_id`, `confirmation_type` (`'age'` или `'sensitive_entry'`), `confirmed_at`.
- Эти записи удаляются при удалении данных пользователя (`delete_user_data`) и при полной очистке (`delete_all_users`).

### Экспорт логов для предоставления по запросу или в суд

- В **админ-панели** добавлена кнопка **«📋 Экспорт логов подтверждений»**. По нажатию бот формирует CSV-файл со всеми записями из `confirmation_logs` и отправляет его админу в чат.
- В файле: `id` записи, `user_id` (Telegram ID пользователя), `confirmation_type`, `confirmed_at` (дата/время в ISO), расшифровка типа (подтверждение возраста / подтверждение перед первой чувствительной записью). Кодировка UTF-8 с BOM для корректного открытия в Excel.
- Сохранённый файл можно при необходимости предоставить по запросу уполномоченных органов или в суд как доказательство факта и времени подтверждения пользователем возраста и согласия на внесение чувствительных данных.
