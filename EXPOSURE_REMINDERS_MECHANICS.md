# ⏰ Механика системы напоминаний дневника экспозиций

**Дата:** 2026-01-27  
**Описание:** Как работает система напоминаний для дневника экспозиций

---

## 📋 ОБЩАЯ КОНЦЕПЦИЯ

Система напоминаний для дневника экспозиций работает по принципу **двухуровневой проверки**:

1. **Планирование через JobQueue** (APScheduler) — для точного времени напоминания
2. **Периодическая проверка** (каждую минуту) — как резервный механизм

---

## 🔄 МЕХАНИКА РАБОТЫ

### 1. **Создание экспозиции и планирование напоминания**

Когда пользователь создает запись экспозиции и указывает **продолжительность события**:

#### Шаг 1: Пользователь указывает продолжительность
- Выбирает из клавиатуры (1, 5, 10, 15... 120 минут) или вводит текстом ("30 минут", "2 часа")
- Продолжительность сохраняется в `exposure_data['event_duration']` (в минутах)

#### Шаг 2: Расчет времени напоминания
```python
event_datetime = "2026-01-27 14:30:00"  # Время начала события
duration_minutes = 30  # Продолжительность события
reminder_datetime = event_datetime + timedelta(minutes=duration_minutes)
# Результат: "2026-01-27 15:00:00" (через 30 минут после начала)
```

#### Шаг 3: Планирование через JobQueue
```python
schedule_exposure_reminder(application, user_id, exposure_id, reminder_datetime)
```

**Что происходит:**
- Вычисляется задержка: `delay = (reminder_datetime - now).total_seconds()`
- Если время уже прошло → напоминание будет отправлено при следующей периодической проверке
- Если время в будущем → создается одноразовая задача через `job_queue.run_once()`
- Задача выполнится точно в `reminder_datetime`

**Пример:**
- Сейчас: `2026-01-27 14:00:00`
- Событие: `2026-01-27 14:30:00`
- Продолжительность: 30 минут
- Напоминание: `2026-01-27 15:00:00`
- Задержка: `3600 секунд` (1 час)
- JobQueue создаст задачу, которая выполнится через 1 час

---

### 2. **Периодическая проверка (резервный механизм)**

Каждую минуту запускается функция `periodic_exposure_reminders()`:

```python
# main.py, строка 202
job_queue.run_repeating(periodic_exposure_reminders, interval=60, first=60)
```

**Что делает `periodic_exposure_reminders()`:**

1. Получает текущее время: `current_datetime = datetime.now().isoformat()`
2. Вызывает `db.get_pending_exposures_for_reminder(current_datetime)`

**SQL запрос в БД:**
```sql
SELECT * FROM exposures
WHERE reminder_sent = 0           -- Напоминание еще не отправлено
AND reality_received = 0          -- Реальность еще не заполнена
AND datetime(event_datetime, '+30 minutes') <= ?  -- Прошло минимум 30 минут после начала
```

**Логика:**
- Если пользователь указал `event_duration` (например, 30 минут), напоминание должно прийти через 30 минут после начала
- Но SQL проверяет `event_datetime + 30 минут` (фиксированные 30 минут)
- Это означает, что напоминание может прийти **раньше**, если `event_duration < 30 минут`
- Или **позже**, если `event_duration > 30 минут` (но не более чем через 30 минут после начала)

**✅ ИСПРАВЛЕНО:** SQL запрос теперь использует `event_duration` вместо фиксированных 30 минут:
```sql
AND datetime(event_datetime, '+' || COALESCE(event_duration, 30) || ' minutes') <= ?
AND event_duration IS NOT NULL  -- Исключаем ручную проработку
```

3. Для каждой найденной экспозиции вызывает `send_exposure_reminder(bot, user_id, exposure_id)`

---

### 3. **Отправка напоминания**

Функция `send_exposure_reminder()` (`exposure_handlers.py`, строка 715):

**Проверки перед отправкой:**
1. ✅ Экспозиция существует и принадлежит пользователю
2. ✅ Реальность еще не заполнена (`reality_received = 0`)
3. ✅ Напоминание еще не отправлено (проверка в БД)

**Действия:**
1. Отмечает напоминание как отправленное: `db.mark_exposure_reminder_sent(exposure_id)`
   - Устанавливает `reminder_sent = 1` в БД
   - Это предотвращает повторную отправку

2. Формирует сообщение с:
   - Названием ситуации
   - Списком ожиданий (если есть)
   - Списком эмоций до события

3. Если есть ожидания:
   - Отправляет сообщение с вопросом "Свершилось ли ваше ожидание?"
   - Кнопки: "✅ Да", "❌ Нет", "⚠️ Не совсем"
   - Устанавливает состояние `WAITING_EXPOSURE_EXPECTATION_FULFILLED`

4. Если ожиданий нет:
   - Отправляет сообщение "Как все прошло на самом деле?"
   - Устанавливает состояние `WAITING_EXPOSURE_REALITY_DESCRIPTION`

## ⚠️ ПРОБЛЕМЫ И ОСОБЕННОСТИ

### Проблема 1: Несоответствие времени напоминания

**Текущая реализация:**
- `schedule_exposure_reminder()` планирует напоминание на `event_datetime + duration_minutes`
- Но `get_pending_exposures_for_reminder()` проверяет `event_datetime + 30 минут` (фиксированные 30 минут)

**Последствия:**
- Если пользователь указал `duration = 15 минут`, JobQueue отправит напоминание через 15 минут
- Но периодическая проверка найдет эту экспозицию только через 30 минут после начала
- Если JobQueue не сработал (перезапуск бота), напоминание придет через 30 минут вместо 15

**✅ ИСПРАВЛЕНО:** SQL запрос теперь использует `event_duration`:
```sql
AND datetime(event_datetime, '+' || COALESCE(event_duration, 30) || ' minutes') <= ?
AND event_duration IS NOT NULL
```

---

### Проблема 2: Ручная проработка

Если пользователь выбрал "Я сам проработаю в удобное для себя время":
- `event_duration = None` (сохраняется в БД как NULL)
- `manual_processing = True` (только в памяти, не сохраняется в БД)
- Напоминание **не планируется** через JobQueue
- Но периодическая проверка все равно найдет эту экспозицию через 30 минут после начала

**Последствия:**
- Пользователь хотел проработать сам, но получит автоматическое напоминание через 30 минут

**✅ ИСПРАВЛЕНО:** Записи с `event_duration IS NULL` теперь исключаются из периодической проверки через условие `AND event_duration IS NOT NULL`.

---

### Проблема 3: Перезапуск бота

Если бот перезапускается:
- Все задачи JobQueue теряются
- Напоминания, запланированные через `run_once()`, не выполнятся
- Остается только периодическая проверка (каждую минуту)

**✅ ИСПРАВЛЕНО:**
- Периодическая проверка работает как резервный механизм
- Теперь она использует `event_duration` вместо фиксированных 30 минут
- Добавлено улучшенное логирование для отслеживания работы системы

```
Пользователь создает экспозицию
    ↓

    ↓
Вычисляется reminder_datetime = event_datetime + duration_minutes
    ↓
┌─────────────────────────────────────┐
│  ДВА ПУТИ ПЛАНИРОВАНИЯ:            │
├─────────────────────────────────────┤
│                                     │

│     ↓                               │
│     schedule_exposure_reminder()    │
│     ↓                               │
│     job_queue.run_once()            │
│     ↓                               │
│     Выполнится в reminder_datetime  │
│                                     │
│  2. Периодическая проверка         │
│     (каждую минуту)                 │
│     ↓                               │
│     periodic_exposure_reminders()   │
│     ↓                               │
│     get_pending_exposures()         │

    ↓

    ↓
mark_exposure_reminder_sent() → reminder_sent = 1
    ↓
Пользователь заполняет реальность
```
